# Decision log — requirements harness rework (genesis)

Thread: docs/threads/260712082750Z-requirements-harness-rework/
Target: seed/seed.md
Subject: settling the shape of the requirements/E2E/BEHAVIOR.md rework — AC granularity, the AC-to-case mapping rule, ID-scheme alignment with jastr, and the handling of non-E2E requirements — before writing the spec.

## P1: What "one-to-one AC-to-case mapping" means, and which case modes it binds

Point: What exactly does "one-to-one" mean — and which case modes does it bind?

What you need to know: There are two directions to pin down. Direction 1: every AC has exactly one covering case (no AC spread across cases, no AC covered five times). Direction 2: every case covers exactly one AC (no case covering seven ACs). Strict bijection means both. The friction is concentrated in two places: (a) contract/scenario mode — `contract-lifecycle` runs real pinned Worktrunk through an imperative multi-step lifecycle and currently proves several ACs (FR-07, FR-13) in one pass; splitting it into one real-Worktrunk run per AC would multiply the slowest suite's runtime and duplicate fragile setup. (b) incidental overlap — a case asserting "init creates every canonical artifact" inherently also demonstrates "init exits 0", so pure bijection forces either artificial blindness (don't assert what you can see) or near-duplicate cases.

A further constraint surfaced during discussion: the living doc's reading experience is the product. With multi-AC cases, an auditor reading BEHAVIOR.md AC-by-AC re-encounters the same case repeatedly without an easy way to tell they already verified it. Also, the fake `wt` shim is not a behavioral mock — wtw only ever spawns `wt --version`; the real dependency direction is Worktrunk invoking wtw through hooks, so fast cases simulate the situations Worktrunk creates, and the contract suite is what validates the assumptions that real Worktrunk actually creates them. Those assumptions are today implicit in one ~300-line scenario test, not enumerated.

Decision: One-to-one everywhere, with the proof unit differing by layer. Fast mode: strict bijection — every AC is covered by exactly one dedicated case, and every case covers exactly one AC. Contract mode: reframed as an explicit Worktrunk-assumptions requirement group — each assumption wtw makes about real Worktrunk becomes its own short AC, and the single real-Worktrunk lifecycle execution declares named checkpoints, each checkpoint covering exactly one AC (the harness gains a checkpoint concept whose checkpoint→AC mapping is enforced by traceability like cases). BEHAVIOR.md renders each AC with its own proof unit's specific evidence (a whole fast case, or one named checkpoint), never a shared multi-AC body. Fast-mode cases assert only their one AC; fixture/assertion duplication across near-identical cases is accepted.

Rationale: The harness and BEHAVIOR.md exist so the owner can audit work delegated to agents; per-AC legibility is the product, and option C is the only shape that provides it in both layers without multiplying real-Worktrunk executions. It also converts the implicit Worktrunk assumptions into an enumerated, auditable list — matching the owner's original two-layer vision (validate assumptions once against real wt, then test fast against fakes). The ripple cost of bijection (a behavior change breaking several near-identical fast cases at once) was flagged and explicitly accepted: agents write and repair the cases, so readability is preferred over maintenance efficiency.

## P2: FR granularity and ID scheme — align with jastr

Point: What FR granularity and ID scheme should wtw adopt to align with jastr?

What you need to know: The two projects differ on three coupled axes, and "align" needs all three pinned. (1) FR granularity: jastr has many narrow FRs — `RUN-FR-0002 — CLI rejects unknown commands` with 2–3 one-line ACs; wtw has one fat FR per category file (13 total), each with 3–6 heavy ACs. The "too few FRs" complaint plus the P1 decision (many short single-assertion ACs) naturally pushes toward jastr's narrow-FR shape — otherwise a fat FR would accumulate 15+ ACs and become a different kind of unreadable. (2) ID prefix: jastr prefixes FRs by domain, matching the manifest file (`RUN-FR-0001` in `01-run.yml`, `ADD-FR-0005` in `15-add.yml`); wtw uses a single project prefix (`WTW-FR-0003`) with global FR numbering across files. (3) AC numbering: jastr numbers ACs locally per FR, restarting at `AC-0001`, and every reference is compound (`RUN-FR-0002.AC-0001`); wtw's AC IDs are globally unique with the FR number baked in (`AC-0301`), which is what lets them "exist on their own" — and it's also the source of the redundancy complaint. With jastr's scheme, a bare `AC-0001` is meaningless out of context, which is fine because the schema can require compound references everywhere. Alignment also implies the statement style: one assertion per statement, no embedded ID, no `(spec AC-03.1)` suffix.

Decision: Full jastr convention, with two codicils added to the spec. Narrow FRs with per-domain prefixes matching the manifest files (e.g. `INIT-FR-0001`, `SYNC-FR-0001`, `CHECK-FR-0001`), ACs numbered locally per FR (`AC-0001`, `AC-0002`, …), and every cross-reference compound (`<FR-ID>.<AC-ID>`). This is a full renumbering: every `covers:` list, the traceability test, and BEHAVIOR.md regenerate; old `AC-0301`-style IDs disappear. Codicil 1 — ID-stability rule: IDs are append-only, never reused, never renumbered; a removed AC/FR keeps its ID with `status: retired`; numbering gaps are normal and meaningless. Codicil 2 — statement grammar: each AC statement is exactly one observable assertion stated as a checkable outcome (exit code, stream content, file state, invocation record), with no embedded IDs and no spec back-references in prose.

Rationale: Cross-project consistency was an explicit goal and jastr is this repo's chartered convention model. The convention also aligns with established practice: narrow FR + one-line ACs mirrors BDD's Feature→Scenario shape (the origin of living documentation), and locally scoped compound AC IDs avoid the "intelligent key" anti-pattern of wtw's FR-encoding global IDs (`AC-0301`), which caused the redundant `(spec AC-03.1)` restatements. ACs are individually addressable for verification but scoped to their FR for meaning — both properties the compound scheme encodes. The codicils close the two gaps the original convention left open: renumbering hazards (the classic failure mode of numbered requirements) and unbounded prose statements.

## P3: Representation of non-E2E-provable requirements

Point: How should non-E2E-provable requirements (today's FR-01, FR-14, and manual-only criteria like the real-Cursor check) be represented in the requirements tree?

What you need to know: Today the requirements tree is deliberately incomplete: FR-01 (workspace architecture, the core/CLI dependency boundary) and FR-14 (the living-doc harness itself) have no manifest because their evidence is unit tests (`dependency-boundary.test.ts`, the schema tests, `docs:living:check`), and the traceability rule only binds FR-02..FR-13. Manual-only criteria follow a third pattern: spec AC-10.4 (a real Cursor visibly opens the workspace) is simply not declared as an AC; its evidence is a human step in `RELEASE-CHECKLIST.md`, mentioned only in a prose `notes:` field. So an auditor reading BEHAVIOR.md sees a tree with invisible holes — you can't distinguish "this requirement doesn't exist" from "this requirement is proven elsewhere". The prose-notes escape hatch is also where the staleness lives (the "Tasks 12 and 13" remnants). Under P1/P2 the tree is being rebuilt anyway, so completeness is cheap to add now and expensive to retrofit later.

Decision: Complete requirements tree with a `verifiedBy` kind on every AC. Every requirement — including today's FR-01/FR-14 material and manual criteria — lives in the manifests. Each AC carries `verifiedBy: case | checkpoint | unit | manual`, and traceability enforces per kind: `case`/`checkpoint` ACs need exactly one covering proof (P1's bijection); `unit` ACs must name the covering test file; `manual` ACs must name the checklist step. BEHAVIOR.md renders everything, visibly labeled by evidence kind. Boundary rule to be stated explicitly in the spec: `unit` is reserved for requirements inherently unreachable by the harness (architecture invariants, the harness itself), never a loophole for skipping E2E on observable CLI behavior.

Rationale: The harness's purpose is a single audit surface for delegated work, and that argument doesn't stop at what the E2E harness can reach — architecture invariants deserve visible, provably covered status as much as CLI behavior. Per-kind enforcement replaces the stale prose `notes:` escape hatch with checkable references, and the `manual` kind formalizes the existing RELEASE-CHECKLIST precedent instead of leaving it as folklore. The cost — a bigger schema and traceability test — is paid once, by an agent.

## P4: Authoritative acceptance registry after the rework

Point: After the rework, what is the authoritative acceptance registry, and does anything still map back to the genesis spec's AC list?

What you need to know: The genesis spec (`docs/threads/260711114414Z-wtw-genesis/specs/001/spec.md`) carries the current FR-01..FR-15 / AC-NN.N list, each citing P-numbers from its decision log, and the manifests back-reference it with the `(spec AC-03.1)` suffixes being eliminated. After P1–P3, the manifests' FRs and ACs will be finer-grained, renumbered, and reorganized — the genesis spec's AC list and the manifests will no longer correspond line-for-line, and under the workflow the genesis spec is a frozen record that can't be rewritten to match. The junior-dev spec for this rework must say what replaces that correspondence, and there's a real risk hiding here: in the rewrite, a behavior the old ACs guaranteed could silently fall out of the new tree, and with the back-references gone nobody would notice.

Decision: The requirement manifests become the single authoritative acceptance registry for observable behavior; the genesis spec's acceptance section becomes historical context. The migration includes a one-time, checked-off mapping (old spec AC → new FR.AC(s) that absorb its substance), living in the rework thread as a migration artifact — used to prove nothing was dropped during the renumbering, then archived with the thread, never maintained afterward.

Rationale: The manifests are already the operationally enforced surface (traceability runs on them, BEHAVIOR.md renders from them); declaring them authoritative acknowledges reality. The one-time audit is the defense against silent behavior loss during the rewrite and costs little; a permanently maintained mapping would be dual bookkeeping that rots, and rewriting the frozen genesis spec would falsify a settled record.

## P5: Cross-project scope — wtw only

Point: Does this rework touch jastr, or is cross-project consistency achieved by wtw adopting the convention wtw-side only?

What you need to know: P2 aligns wtw to jastr's existing format, so the ID-scheme inconsistency disappears without touching jastr. But P1–P3 also extend the convention beyond what jastr has today: scenario checkpoints, the `verifiedBy` kinds, the ID-stability rule, the one-assertion grammar. After this rework, wtw will briefly be the more advanced convention — so "consistency" has a direction question: does anything here obligate changes in the jastr repo?

Decision: This thread and its spec are scoped to wtw only; jastr is not touched — it was a reference, nothing more. The spec must state the convention fully and self-containedly, never referencing the jastr repo as a build dependency (jastr remains inspiration, not authority; `.library/sources/` is gitignored and may be absent on a fresh clone). Back-porting the extensions (checkpoints, `verifiedBy`, codicils) to jastr, if ever desired, is a separate future thread in the jastr repo.

Rationale: The working target is wtw; jastr served as the convention model, and importing its format does not create an obligation to export refinements back. Keeping the spec self-contained protects the junior-dev handoff from depending on an absent gitignored reference tree. The previously rejected 1:1 proposal in jastr, if revisited, belongs in that project's own thread with this project as working proof.
